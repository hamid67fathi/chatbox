مشتری تایپ می‌کند → Widget [Vanilla JS] WebSocket connect به WS Gateway (Go) → WS Gateway پیام دریافت → Chat Service (Node.js) gRPC call → Chat Service: validation → DB write (PostgreSQL) → Redis Pub/Sub event publish: "conversation:{id}:message" → تمام WS Serverها subscribe به Redis → پیام به اپراتور آنلاین push می‌شود (زیر ۵۰ms) اگر اپراتور آفلاین: → BullMQ job → Notification Service → Push notification (FCM / APNs) + Email موازی — بدون تأخیر روی پیام اصلی: → Bot Engine: آنالیز پیام → پیشنهاد پاسخ (AI Copilot) → Analytics Service: event log → ClickHouse (async، non-blocking)
🏢

Multi-Tenancy Strategy

  • Schema-per-tenant در PostgreSQL تا ۱۰۰۰ workspace
  • Row-level Security (RLS) با tenant_id برای مقیاس بزرگ‌تر
  • جداسازی Redis با prefix: ws:{tenant_id}:*
  • S3/MinIO: جداسازی bucket یا prefix per tenant
  • Rate limiting per tenant در API Gateway
  • هر workspace سقف مشخص CPU، RAM، Storage دارد
📈

مقیاس‌پذیری افقی

  • WS Gateway (Go): کاملاً stateless، horizontal scale
  • HorizontalPodAutoscaler در K8s بر اساس WS connection count
  • PgBouncer: Connection pooling جلوی PostgreSQL
  • Redis Cluster: sharding خودکار برای مقیاس بالا
  • CDN ArvanCloud: ویجت JS از edge، بدون origin hit
  • Kafka در فاز ۴: جایگزین Redis Pub/Sub برای ۱M+ msg/s

۳ · ریل‌تایم

Real-time Engine — قلب سیستم

بحرانی‌ترین بخش فنی کل پروژه. هر باگ اینجا مستقیم مشتری می‌بیند.

🚨 هشدار: این بخش را از ابتدا درست طراحی کنید. Migration از polling به WebSocket بعد از launch هزینه بسیار بالایی دارد و همه مشتریان را تحت‌تأثیر می‌گذارد.
🔌

WebSocket Architecture

  • WS Gateway با Go — هدف: ۵۰K concurrent روی یک سرور ۱۶ core
  • هر connection یک goroutine سبک — نه thread سنگین OS
  • Heartbeat هر ۳۰ ثانیه — تشخیص connection مرده
  • Reconnection با exponential backoff در Client
  • Socket.io Redis Adapter — همه serverها sync هستند
  • Sticky sessions در Load Balancer (IP hash)
📬

Message Delivery Guarantee

  • هر پیام یک UUID دارد — idempotency key
  • Client ACK — سرور منتظر تأیید است
  • Retry logic: ۳ بار، سپس fallback به HTTP polling
  • Offline queue: پیام‌های فرستاده‌نشده در IndexedDB
  • Read receipt دو مرحله‌ای: Delivered + Seen
  • At-least-once delivery با deduplication در Chat Service

🏠 Room/Channel Strategy

// هر مکالمه یک Room مجزاست socket.join(`conversation:${conversationId}`) // هر workspace channel خودش را دارد socket.join(`workspace:${workspaceId}`) // اپراتورها channel اختصاصی دارند socket.join(`agent:${agentId}`) // رویدادهای اصلی — server → client: 'message:new' // پیام جدید از مشتری 'message:seen' // مشتری پیام را دید 'typing:start' // تایپینگ indicator 'typing:stop' // توقف تایپ 'agent:assigned' // اپراتور به مکالمه اضافه شد 'conversation:close' // مکالمه بسته شد 'visitor:online' // بازدیدکننده آنلاین شد
⌨️

Typing Indicator — جزئیات پیاده‌سازی

  • Debounce 300ms قبل از ارسال typing:start
  • Timeout خودکار ۳ ثانیه برای typing:stop
  • فقط به اپراتور مربوطه آن مکالمه ارسال می‌شود
  • در Redis با TTL 3s — هیچ‌وقت به DB نمی‌نشیند
  • در فرم آفلاین نمایش داده نمی‌شود
🚧

Scaling Bottlenecks — باید بدانید

  • Redis Pub/Sub در ۱۰۰K+ msg/s به bottleneck می‌رسد — فاز ۴: Kafka
  • Elasticsearch write lag در spike — async indexing با buffer
  • Presence (آنلاین/آفلاین) — TTL-based در Redis، نه event
  • File upload به WS نرود — مستقیم presigned URL به MinIO
  • AI response زمان‌بر است — streaming SSE برای Copilot

-- هر کسب‌وکار یک Workspace است workspaces: id, slug, name, plan, settings, created_at -- اپراتورهای داشبورد agents: id, workspace_id, email, role, status, last_seen, settings -- بازدیدکنندگان سایت‌های مشتریان contacts: id, workspace_id, visitor_id, name, email, phone, meta, created_at -- مکالمات conversations: id, workspace_id, contact_id, agent_id, channel, status, priority, tags[], first_reply_at -- پیام‌ها — بزرگ‌ترین جدول messages: id, conversation_id, sender_type, sender_id, content, type, attachments[], status, created_at -- Knowledge Base برای AI kb_articles: id, workspace_id, title, content, embedding_id, published -- لاگ همه اکشن‌های admin audit_logs: id, workspace_id, actor_id, action, resource_type, meta, ip

⚡ Critical Indexes

-- مکالمه‌های یک workspace — متداول‌ترین query CREATE INDEX idx_conv_workspace_status ON conversations(workspace_id, status, created_at DESC); -- پیام‌های یک مکالمه (pagination) CREATE INDEX idx_msg_conversation_time ON messages(conversation_id, created_at DESC); -- جستجوی contact با ایمیل CREATE INDEX idx_contact_workspace_email ON contacts(workspace_id, email); -- Visitor lookup (بدون کوکی) CREATE INDEX idx_contact_visitor_id ON contacts(visitor_id) WHERE visitor_id IS NOT NULL; -- Full-text search فارسی (backup برای ES) CREATE INDEX idx_msg_fts ON messages USING GIN(to_tsvector('simple', content));
💡 همیشه EXPLAIN ANALYZE را روی query‌های production اجرا کنید. N+1 query در لود بالا سیستم را از پا در می‌آورد.

📊 ClickHouse — Analytics Schema

-- رویدادهای time-series (فقط insert — نه update) CREATE TABLE events ( workspace_id UUID, event_type String, -- 'message'، 'conversation_open' conversation_id UUID, agent_id UUID, visitor_id String, value Float64, -- زمان پاسخ، CSAT score meta String, -- JSON metadata created_at DateTime ) ENGINE = MergeTree() PARTITION BY toYYYYMM(created_at) ORDER BY (workspace_id, created_at) TTL created_at + INTERVAL 2 YEAR;

📦 Migration Strategy — Zero Downtime

۱. Prisma Migrate: schema version control — هر تغییر یک migration file مجزا.
۲. Backward Compatible: هیچ column را مستقیم حذف یا rename نکنید. ابتدا column جدید اضافه کنید، در release بعد قدیمی را حذف کنید.
۳. Read Replica: یک replica جداگانه فقط برای dashboard queries و analytics — بار از primary کم می‌شود.
۴. WAL Archiving: Point-in-time recovery با حداقل ۷ روز undo. اشتباهات را می‌توانید برگردانید.

۶ · امنیت

امنیت لایه‌به‌لایه

امنیت باید از روز اول در architecture باشد — نه بعداً اضافه شود.

🛡️ اصل اساسی: هر tenant باید کاملاً ایزوله باشد. یک tenant نباید بتواند داده tenant دیگر را ببیند — حتی در صورت باگ در کد. این با RLS در PostgreSQL و prefix جداگانه در Redis تضمین می‌شود.
🔑

Authentication & Authorization

  • JWT با access token کوتاه (۱۵ دقیقه) + refresh token (۷ روز)
  • Refresh token rotation — هر refresh یک token کاملاً جدید
  • Token revocation list در Redis (برای logout فوری)
  • OAuth2 / Social login برای agents
  • SAML/SSO برای Enterprise clients
  • RBAC: owner > admin > agent > viewer
  • 2FA اجباری برای role admin و owner
🛡️

API Security

  • Rate limiting: ۱۰۰ req/min per IP (widget)، ۱۰۰۰ برای authenticated
  • CORS: فقط domain‌های ثبت‌شده workspace مجاز هستند
  • Webhook signature verification با HMAC-SHA256
  • Input validation با Zod در هر endpoint — بدون exception
  • SQL injection: Prisma ORM — هیچ raw query بدون parameterize
  • Content Security Policy header در داشبورد
🔐

Data Encryption

  • TLS 1.3 برای همه traffic: HTTPS + WSS
  • داده حساس (ایمیل، شماره، پیام‌های حساس) با AES-256
  • Encryption at rest در PostgreSQL
  • MinIO: server-side encryption برای فایل‌ها
  • Environment secrets در K8s Secrets (HashiCorp Vault برای بلندمدت)
  • Backup‌ها رمزنگاری‌شده قبل از ذخیره در object storage
🧱

Widget Isolation — بحرانی

  • ویجت در iframe sandbox — دسترسی به DOM سایت اصلی ندارد
  • postMessage API برای ارتباط ایمن بین ویجت و سایت
  • هیچ API key در client-side نیست — فقط workspace_token عمومی
  • SRI (Subresource Integrity) برای script tag ویجت
  • CSP header از سمت ویجت JS
  • DDoS protection از طریق ArvanCloud WAF

⚠️ ریسک‌های فنی اصلی

  • پیچیدگی Real-time در مقیاس بزرگ — نادیده نگیرید
  • Multi-tenancy اگر از ابتدا درست نباشد، Migration بسیار دردناک است
  • هزینه AI API با دلار — بدون سقف می‌تواند bankruptcy ایجاد کند
  • WebSocket و فیلترینگ — همیشه fallback داشته باشید
  • Widget conflict با سایت مشتری — sandbox اجباری است
  • Elasticsearch در مقیاس — expensive است، بعداً اضافه کنید

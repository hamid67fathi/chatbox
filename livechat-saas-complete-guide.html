فایل و رسانه

🔄 جریان پیام ریل‌تایم

چطور یک پیام از مشتری به اپراتور می‌رسد:
۱. کاربر پیام می‌فرستد → Widget از طریق WebSocket به Gateway وصل است
۲. Gateway پیام را دریافت → Chat Service صدا می‌زند
۳. Chat Service پیام را در PostgreSQL ذخیره → رویداد را در Redis Pub/Sub منتشر می‌کند
۴. تمام WebSocket serverها به Redis subscribe هستند → پیام به اپراتور آنلاین push می‌شود
۵. اگر اپراتور آفلاین → Notification Service پوش نوتیف / ایمیل می‌فرستد

📈 مقیاس‌پذیری

  • WebSocket serverها Stateless (با Redis Adapter)
  • Horizontal scaling با Docker + Kubernetes
  • Connection Pooling در دیتابیس
  • CDN برای ویجت (کمترین تأخیر لود)
  • Queue برای ارسال نوتیفیکیشن‌های انبوه

🔒 امنیت

  • End-to-End Encryption انتخابی
  • JWT با refresh token rotation
  • Rate limiting در API Gateway
  • DDoS protection (Cloudflare)
  • رمزنگاری داده حساس در دیتابیس
  • Audit Log برای همه اکشن‌های مدیران

۷ · MVP

MVP — محصول اولیه قابل عرضه

با چه چیزی شروع کنید؟ فقط همین‌ها — نه بیشتر.

✅ قانون MVP: باید بتوان در کمتر از ۳ ماه به بازار رسید. هر چیزی که این rule را نقض کند، به Phase 2 منتقل می‌شود.

✅ در MVP هست

  • چت ریل‌تایم با WebSocket
  • ویجت قابل نصب (کد JS)
  • داشبورد اپراتور (وب)
  • مدیریت چندین اپراتور
  • تاریخچه مکالمه (۳۰ روز)
  • پلن رایگان + ۱ پلن پولی
  • اعلان مرورگر
  • فرم آفلاین + ایمیل
  • پشتیبانی RTL فارسی
  • موبایل پاسخگو (Responsive)
  • اپ موبایل اپراتور (اولیه)
  • آمار پایه (تعداد چت، میانگین پاسخ)

🚫 در MVP نیست

  • AI Chatbot (Phase 2)
  • Automation / Workflow
  • واتساپ/تلگرام
  • Video Call
  • Knowledge Base
  • Ticketing
  • Co-browsing
  • Marketplace
  • Analytics پیشرفته
  • White-label
  • API عمومی

هوش مصنوعی

  • هزینه‌های API با دلار — مدیریت ریسک ارزی

واتساپ و کانال‌ها

  • Business API واتساپ نیاز به شماره خارجی
  • تلگرام جایگزین اصلی — اولویت داشته باشد
  • ایتا، روبیکا — کانال‌های ایرانی جایگزین
